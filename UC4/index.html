02
Claude.ai · Claude Code · Répartition des rôles

Le bon outil
pour la
bonne tâche

Claude.ai pour le jugement : cadrer, choisir, arbitrer. Claude Code pour l’exécution : intégrer, harmoniser, déployer. La règle appliquée : tâche de décision → Claude.ai, tâche d’implémentation → Claude Code. Confondre les deux, c’est obtenir des réponses techniques à des questions éditoriales.

Claude.ai
Cadrage
Définir le scope des UC, arbitrer la DA, valider chaque étape éditoriale. Les décisions restent humaines.
Claude.ai
Validation
Relecture de chaque contenu, correction des marqueurs IA, arbitrage des formulations. L’œil critique avant livraison.
Claude Code
Assemblage
Centralisation CSS (~720 lignes inline → 1 fichier), migration design system, intégration des UC, cohérence cross-pages.
Claude Code
Déploiement
Build Netlify, vérification des liens, tests cross-device, maintenance continue. L’exécution sans friction.
Ce que ça révèle
Utiliser deux outils IA sans définir leurs rôles, c’est comme travailler avec deux PO sur le même backlog. La répartition des rôles précède l’exécution.

01
4 UC ciblés vs Couvrir plus de terrain
Chaque UC supplémentaire sans usage réel dilue la crédibilité. Les 4 UC retenus correspondent chacun à une pratique effective : méthodes utilisées quotidiennement, agent actif sur chaque candidature, veille publiée chaque lundi, portfolio construit avec les outils qu’il documente.
Sans ce choix : un portfolio plus large mais moins crédible : des cas d’usage sans trace d’usage réel.
02
Tabs (1 case à la fois) vs Scroll vertical (tout visible)
Les tabs forcent la lecture séquentielle : le recruteur lit un case study complet avant de passer au suivant. Le scroll permettait de voir les 4 cases d’un coup, mais ça invite à survoler plutôt qu’à lire. Décision d’architecture de l’information, pas de UI.
Sans ce choix : une page trop longue où chaque case study est survolé plutôt que lu.
03
styles.css centralisé vs CSS inline dupliqué ×3
Une seule source de vérité prévient les divergences, comme un backlog unifié évite les priorités contradictoires. 3 fichiers HTML avec chacun ~240 lignes de CSS inline signifiait 3 interventions manuelles à chaque modification de palette ou de typo.
Sans ce choix : des incohérences visuelles accumulées à mesure que le portfolio grandit, impossibles à rattraper sans tout reprendre.
Ce que ça révèle
Documenter ce qu’on n’a pas fait, et expliquer pourquoi, est aussi utile que documenter ce qu’on a livré. Un choix sans rejet explicité, c’est une préférence, pas une décision.

04
Vue d’ensemble · Cohérence · Progression

Un système,
pas
une liste

Les 4 UC suivent une progression : méthodes de base, agent de préparation, automatisation, et le portfolio comme preuve finale. Chaque couche s’appuie sur la précédente. UC2 utilise les méthodes d’UC1. UC3 a été construit avec l’Agent Coach d’UC2. UC4 documente tout en temps réel avec les mêmes outils.

UC1 Structurer un prompt, tester sur de vrais documents, mesurer les écarts entre modèles.
UC2 Concevoir un agent A–Z : architecture, RAG, modes distincts, usage terrain sur chaque candidature.
UC3 Automatiser avec Python et APIs brutes, sans no-code. Agent fonctionnel en une journée.
UC4 Orchestrer deux outils IA sur un livrable complet, documenter chaque décision en temps réel.
Ce que ça révèle
La progression n’était pas planifiée dès le départ. Elle s’est révélée en construisant. C’est souvent comme ça que fonctionnent les bons systèmes produit.
